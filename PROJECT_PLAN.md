Architectural Blueprint for Open Source Smart Mirror Platform

Project Vision and Core Philosophy
The objective of this project is to build an open source, highly modular smart mirror platform designed to execute efficiently on low power, headless hardware such as a Raspberry Pi utilizing an external SSD. To maximize community involvement, the entire ecosystem is built using JavaScript and TypeScript, lowering the barrier to entry for frontend and full stack hobbyists. The framework intentionally avoids heavy server side rendering frameworks like Next.js to preserve CPU and RAM on embedded machinery, focusing instead on high performance execution and plug and play module ergonomics.

Execution and Runtime Environments
The application leverages a dual runtime strategy to optimize the experience for both active developers and production deployments. For local testing and module creation, the system runs on standard Node.js and TypeScript toolchains with hot module replacement enabled, allowing creators to write code via remote workflows like VS Code SSH directly on the Pi. In a live production environment, the platform is targeted to run on Bun. Bun serves as a high performance, drop in replacement for Node.js that dramatically cuts down memory footprint, ensures instantaneous boot times, and provides native, fast implementations of SQLite and WebSockets.

Backend Architecture and Dynamic Routing
The core server is constructed on Fastify due to its low overhead and robust plugin encapsulation model. The server is responsible for serving the static frontend dashboard, maintaining an internal SQLite database layer via an abstraction tool like Kysely, and running a real time event pipeline using WebSockets to push live telemetry, time updates, and touchscreen actions across the application.

At boot time, the Fastify instance runs a bootstrap script that scans a dedicated modules directory on the storage drive. Each module folder contains its own configuration metadata, a frontend component, and an optional backend file. The core engine dynamically mounts each module backend using Fastify's native registration loops. To prevent third party code from breaking the main application, all custom plugin endpoints are strictly sandboxed under a scoped URL prefix based on the module directory name.

Frontend and UI Foundations
The user interface is designed as a single page application powered by React, TypeScript, and Vite. For styling, the system combines Tailwind CSS with daisyUI as a foundational Tailwind plugin. Complex interactive states that require accessible keyboard or touch controls, such as modals or dropdown overlays, leverage headless Radix UI primitives following the popular shadcn design philosophy. However, all visual presentation layers are built entirely using semantic daisyUI utility classes. This class-based component layer ensures that styling selectors exist globally within the master build pipeline, allowing third party plugins to use standard components without risking having their utility classes stripped out by Tailwind's compilation purging steps. Global theme synchronization is handled through the native HTML data-theme attribute, ensuring all widgets instantly match the user's aesthetic preferences.

Exploring AI Integration and Dynamic UI Generation
A standout feature of this platform is a built in AI configuration layer allowing users to supply their Claude API credentials. Beyond simple chat capabilities on the touchscreen, this allows the mirror to dynamically generate functional interfaces on the fly based on verbal or typed user prompts. Two architectural paths are proposed to handle this real time interface injection.

The first approach is the Client Side JSX Runtime Compiler. When a user requests a custom widget, Claude evaluates the prompt and returns a clean string of standard React JSX using daisyUI classes. The Fastify backend intercepts this string and pushes it instantly to the frontend SPA via WebSockets. The browser utilizes a lightweight, assembly based runtime compiler like Babel Standalone or SWC WASM Web to compile the string into an executable component directly in browser memory, which is then mounted inside an AI layout container. This path requires zero backend compilation overhead, prevents storage drive wear, and instantly integrates with the master stylesheet because daisyUI class names are already loaded globally.

The second approach is the Iframe Sandbox Template. This strategy acknowledges that artificial intelligence excels at taking quick structural shortcuts. Instead of generating complex React code that risks crashing the main dashboard if a bracket is missed, Claude outputs a standard, standalone static HTML block using vanilla JavaScript and daisyUI classes. The Fastify backend saves this text file directly to the external SSD into an AI module folder. The main React dashboard then loads this code inside a standard iframe. To maintain visual consistency, the backend automatically wraps Claude's raw code in a boilerplate template that links to the daisyUI CDN and copies the mirror's current theme into the iframe's HTML header. Because the file is hosted locally on the same origin port, the inline JavaScript can safely make standard fetch calls back to the Fastify API to control smart devices or pull information. If the AI hallucinates bad code, the error is perfectly sandboxed within the iframe, guaranteeing the core system remains operational.
